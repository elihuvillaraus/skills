# Always On Memory Skill

## What This Skill Does

The **Always On Memory** skill maintains persistent, structured documentation throughout architect and orchestrator execution. It creates **five key files** that capture context, preferences, and actions in real-time.

### File Hierarchy

```
project/
  AGENTS.md          ← Hub (≤150 lines). Auto-read by all AIs. Points to everything.
  memory.md          ← AI-maintained preferences. Updated every session.
  context/
    pipeline.md      ← Full Pipeline Laws (generated by /project-init)
    tech-stack.md    ← Stack + conventions (generated by /project-init)
  docs/
    ALWAYS-ON-MEMORY.md  ← Session decisions, learnings, blockers
    USER-QA.md           ← Manual QA checklist
    USER-TASKS.md        ← User action items
```

> **Bootstrap rule**: If `AGENTS.md` or `context/` are missing → run `/project-init` before proceeding.

This skill ensures that:
- **AGENTS.md exists** and stays short (≤150 lines) — the universal anchor for all AI tools
- **memory.md is updated** at end of every session with learned preferences
- **Decisions are documented** as they're made (architect phase)
- **Learnings are captured** from implementation (orchestrator/ralph phase)
- **QA requirements are clear** for the user (generated in parallel with tester)
- **User tasks are identified** automatically and tracked (generated in parallel with tester)

## Core Responsibility

This skill is **automatically loaded** by `architect` and `orchestrator` skills. It provides:

1. **Bootstrap check** — ensures AGENTS.md + context/ + memory.md exist (run /project-init if not)
2. **Template generation** for session documents
3. **Real-time updates** as work progresses
4. **memory.md maintenance** — update at end of every session with new learnings
5. **Parallel document generation** (doesn't block implementation)
6. **Auto-detection of user tasks** during planning and execution
7. **USER-JOURNEY sync** (updates if needed during execution)

## Phase 0 — Bootstrap Check

Before doing anything else:

```bash
# Check if 4-file system is set up
ls AGENTS.md context/pipeline.md memory.md 2>/dev/null
```

- **All found** → continue
- **Missing** → say: "AGENTS.md or context/ not found. Run `/project-init` to set up the AI memory system for this project. Continue anyway? (y/n)"
- **User says y** → continue without them; note missing files in ALWAYS-ON-MEMORY.md

## memory.md — The Preference Log

`memory.md` is the AI's long-term preference memory. It's different from `ALWAYS-ON-MEMORY.md`:

| | `memory.md` | `docs/ALWAYS-ON-MEMORY.md` |
|--|-------------|---------------------------|
| What | Preferences + corrections | Decisions + learnings per session |
| Who writes | AI (auto, every session) | AI (auto, per session) |
| Grows | Slowly (only substantial items) | Per session |
| Survives | All sessions | Per session |

**When to update `memory.md`**:
- User corrects a recurring mistake → add to "Corrections Log"
- User states a preference ("never use X", "always Y") → add to relevant section
- A pattern emerges that saves time → add to "Workflow Preferences"

**What NOT to add**:
- One-off decisions
- Session-specific context
- Things already in `context/pipeline.md` or `context/tech-stack.md`

**At end of every session**: review what was learned and update `memory.md` in-place. Replace outdated entries.

## Integration Points

### With `architect` Skill
```
1. Load always-on-memory skill
2. Create docs/ALWAYS-ON-MEMORY.md with Session Info + Key Decisions section
3. As PRD is built:
   - Document architectural decisions in ALWAYS-ON-MEMORY.md
   - Identify user tasks in USER-TASKS.md (from PRD requirements)
4. Create skeleton of USER-QA.md based on USER-JOURNEY
```

### With `orchestrator` Skill
```
1. Load always-on-memory skill
2. Start ralph parallel implementation
3. Start tester parallel testing
4. Continuously update ALWAYS-ON-MEMORY.md:
   - Log learnings from ralph
   - Log blockers and solutions
   - Update with tester insights
5. Generate full USER-QA.md (tester provides detailed test results)
6. Identify additional USER-TASKS during ralph execution
7. DO NOT WAIT for user to complete tasks - continue work
```

### With `tester` Skill
```
1. Run all automated tests (unit, integration, E2E)
2. Generate detailed USER-QA.md with:
   - Each USER-JOURNEY step with expected vs actual
   - Critical issues highlighted
   - Verification steps for user
3. Do this IN PARALLEL - don't wait for user QA
```

## Auto-Detection of User Tasks

**During Planning (architect phase)**:
- Look for patterns like:
  - "requires API key"
  - "user must configure"
  - "external service setup"
  - "deployment step"
  - "manual approval"
  
**During Execution (orchestrator phase)**:
- Ralph or tester discover tasks like:
  - "Need database credentials"
  - "Need to setup webhook"
  - "Need to configure payment provider"
  - "Manual deployment required"

**Detection Logic**:
1. Scan PRD for user-required actions
2. When ralph encounters external service → log as USER-TASK
3. When tester encounters manual step → log as USER-TASK
4. Continuously update USER-TASKS.md with new discoveries

---

## USER-JOURNEY Synchronization

If during execution the USER-JOURNEY needs updates:
1. Identify what's missing or incorrect
2. Update `docs/USER-JOURNEY.md` with new insights
3. Update section refs in USER-QA.md
4. Document change in ALWAYS-ON-MEMORY.md under "Implementation Notes"

## Best Practices

1. **ALWAYS-ON-MEMORY is your north star**
   - Every decision should be traceable
   - Every blocker should be documented
   - Every learning should be captured

2. **USER-QA is not blocking**
   - User performs this AFTER all automated tests pass
   - If issues are found, they go back to development
   - This is the final human touch before production

3. **USER-TASKS don't block work**
   - Implementation continues regardless
   - Tasks are prioritized by blocker status
   - User completes in parallel with development

4. **Keep docs updated in real-time**
   - Don't batch updates
   - Capture insights immediately
   - This becomes your institutional knowledge

5. **If USER-JOURNEY changes, document it**
   - Note why it changed
   - Update all dependent documents
   - Communicate impact to team
